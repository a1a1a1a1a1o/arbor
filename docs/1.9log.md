## A) Release blockers (must fix before 1.9.0)

### Blocker matrix

| ID | Area | Risk | Shipping impact | Target |
|---|---|---|---|---|
| A1 | Version consistency | High | Incorrect release identity across ecosystem tooling | 1.9.0 |
| A2 | Graph index safety | Critical | Potential graph corruption under incremental updates | 1.9.0 |
| A3 | C# parser parity | High | Language support mismatch and silent omissions | 1.9.0 |
| A4 | Sync server hardening | Critical | Metadata exposure in networked/headless mode | 1.9.0 |

### A1. 🚫 Version consistency across all artifacts

Current state is inconsistent:

- Workspace and crate manifests still reference older versions
- VS Code extension metadata not aligned with 1.9.0
- Packaging manifests (Homebrew/Scoop/npm) not all aligned
- Watermark and protocol hello/version values hardcoded in app/server paths

Required outcome:

- Every external-facing version string reports **1.9.0** at release cut.

### A2. 🚫 Graph node index safety for incremental removals

Issue:

- Node removal semantics can invalidate indices used by lookup maps.

Risk:

- Silent graph corruption after incremental updates.

Required outcome:

- Stable node identity guarantees during remove/update flows (or explicit full-rebuild policy).

### A3. 🚫 Legacy C# parser parity

Issue:

- C# support differs between parser paths.

Risk:

- Files silently omitted depending on code path.

Required outcome:

- Consistent C# behavior across supported parser flows.

### A4. 🚫 Sync server exposure hardening

Issue:

- Headless/network deployments can expose graph metadata without robust access controls.

Risk:

- Unauthorized metadata access.

Required outcome:

- Restrictive defaults (localhost), plus origin/token controls for non-local operation.

## B) High-priority correctness fixes

1. ⏳ Caller/callee analysis should not be limited to only one edge kind where broader dependency semantics are intended.
2. ⏳ Dependents traversal must use true BFS where hop distance is part of output.
3. ⏳ Graph export must include edges, not just nodes.
4. ⏳ Language capability response must be generated from actual supported language registry.
5. ⏳ Edge-kind propagation in builder must preserve relationship semantics (not collapse all to Calls).
6. ⏳ Production `.unwrap()` usage in graph/sync export paths should be replaced by safe error handling.

### Correctness matrix

| ID | Summary | Risk | Value |
|---|---|---|---|
| B1 | Multi-edge-kind dependency semantics | High | Accurate impact analysis |
| B2 | BFS dependents traversal | Medium | Correct hop-distance reporting |
| B3 | Include edges in graph export | Medium | Usable exports for external consumers |
| B4 | Dynamic language capability response | Medium | Truthful API contract |
| B5 | Preserve edge-kind semantics in builder | High | Graph relationship integrity |
| B6 | Replace panic-prone unwrap paths | High | Runtime resilience |

---

## C) Performance targets

1. ⏳ Parallelize parse/index pipeline where thread-safe.
2. ⏳ Replace O(n) fallback scans in search/symbol resolution with reverse indices.
3. ⏳ Unify and improve token estimation using real source-aware approach.
4. ⏳ Replace full periodic reindex in watch mode with file-event driven delta updates.
5. ⏳ Move large broadcast updates from full graph snapshots to deltas.
6. ⏳ Optimize centrality path with adjacency precomputation and direct graph traversal APIs.

### Performance matrix

| ID | Summary | Expected gain |
|---|---|---|
| C1 | Parallel parse/index | 2x–4x faster indexing on multi-core hosts |
| C2 | Search/symbol reverse index | Lower query latency on large graphs |
| C3 | Source-aware token estimation | Better context budgeting reliability |
| C4 | Event-driven watch updates | Lower idle CPU + less unnecessary IO |
| C5 | Delta-only sync broadcasts | Lower memory/bandwidth overhead |
| C6 | Centrality adjacency optimization | Faster ranking on large graphs |

## D) Architecture and maintainability

1. ⏳ Split large CLI command monolith into command-group modules.
2. ⏳ Converge parser systems (reduce v1/v2 behavioral divergence).
3. ⏳ Deduplicate extension/watch lists to a single source of truth.
4. ⏳ Remove stale/misleading comments and dead fields/variables.
5. ⏳ Standardize error handling strategy across crates.
